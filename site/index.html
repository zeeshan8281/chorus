F-01

SHARED NAMESPACES

One memory per team, project, or session. Reads stay scoped — no cross-tenant leakage.

F-02

UPSERT BY KEY

Write to a key and the shared fact updates in place. One source of truth per thing, not a pile of copies.

F-03

CONTRIBUTOR TRACKING

Every entry records who wrote the current value and everyone who's touched it, with a revision counter.

F-04

ACTIVITY FEED

Agents catch up on what changed since their cursor — back in sync without re-reading everything.

F-05

RECALL & FORGET

Read by key, text, or tag. Retract a fact the team no longer trusts. Simple, fast, predictable.

F-06

TS-NATIVE · RAILWAY-READY

Library or HTTP service. Node 22+, zero-config deploy on Railway with file-backed durability.

FAQ <GO> · STRAIGHT ANSWERS HELP HELP
Q. What problem does CHORUS actually solve?

Coordination. When several agents work together they each keep their own siloed memory, re-learn the same context, and drift apart. CHORUS gives them one shared memory so they stay on the same page — write once, everyone benefits, everyone can catch up.

Q. What happens when two agents write the same key?

It's an upsert: the shared fact updates in place, revision bumps, and both agents are recorded as contributors. You see what it replaced (previousContent) — no duplicate entries, one source of truth per key.

Q. How do agents stay in sync?

Each agent keeps a cursor (the last activity seq it saw) and calls activity(namespace, since) to pull only what changed. No re-reading the whole store — just the delta since it last looked.

Q. Does it replace Mem0 / Zep / my vector DB?

It's the shared-coordination layer on top. The semantics live above a storage interface, so an in-memory, file, or Postgres backend (or a vector DB) can sit underneath.

Q. Where does it run?

Anywhere Node 22+ runs — as a library or an HTTP service. Deploys on Railway with the included config and file-backed durability; swap in Postgres for multi-instance.
